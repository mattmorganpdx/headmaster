URL filter help

A rule's URL filter decides which requests it applies to. It's matched against the request's full URL using Chrome's declarativeNetRequest pattern syntax (the same style as ad-blocker filter lists) — not a regular expression.

Special characters

Token Meaning Example
|| Domain anchor. Matches a host and its subdomains, on any scheme (http/https), anchored at a domain boundary. This is the usual way to target a site. ||dev.example.com
| Start/end anchor. At the beginning or end of the pattern, pins it to the start or end of the whole URL. |https://dev.
* Wildcard. Any run of characters. */api/*
^ Separator. Matches a separator character (end of a host or path segment) or the end of the URL. ||example.com^

A filter with no special characters is a plain substring match — it matches any URL that contains that text anywhere. That's looser and can over-match, so prefer || when you mean "this host".

What ||dev.example.com matches

✓ https://dev.example.com/anything

✓ http://dev.example.com

✓ https://api.dev.example.com (a subdomain)

✗ https://example.com (the parent, not dev)

✗ https://mydev.example.com (no real domain boundary)

The headline use case

Because every rule is independent, the same header name can carry different values per environment, all active at once:

Header Value URL filter
X-Env dev ||dev.example.com
X-Env staging ||staging.example.com
X-Env prod ||example.com

Site access

When you enable a rule, Headmaster asks for access to just the host its filter targets — so it can only change headers on sites you've explicitly allowed. A filter that can't be tied to one host (a path-only or substring pattern) asks for broader access instead.
